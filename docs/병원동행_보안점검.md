- **작성일**: 2026-07-16
- **작성자**: 조경호
- **대상**: https://nike1137-svg.github.io/hospital-app/ (프론트) + Supabase (백엔드)
- **목적**: 제출 전 보안 스스로 점검 (루브릭 ① 대응). "점검했다"에서 끝나지 않고 **실제로 문제가 없는지 검증**한 결과를 기록.

## 1. 과제 공통 보안 체크리스트

| 항목 | 결과 | 근거 |
|------|:---:|------|
| 비밀값(키)을 코드에 직접 안 쓰고 환경변수로 뺐는가 | ✅ | `.env.local` 사용, 소스에 키 하드코딩 없음 (아래 2번 검증) |

| (Supabase) RLS 켜고 정책 넣었는가 | ✅ | 라이브 DB 실제 테스트로 확인 (아래 3번) |
| 로그인 필요 데이터에 권한 확인이 되는가 | ✅ | 개인정보(연락처) 담긴 `requests`는 조회 정책 없음 → 공개 조회 차단 |

## 2. 비밀값 관리 — 실제 검증

**규칙**: 코드 = 공개 OK / 비밀값 = 환경변수. 정적 사이트라 `NEXT_PUBLIC_` 값은 브라우저에 노출되므로 **공개 안전한 `publishable` 키만 사용**, `service_role`(마스터) 키는 프론트에 절대 미사용.

검증 명령과 결과:
- `git ls-files | grep .env` → **`.env` 추적 안 됨** ✅
- `git grep "sb_publishable|sb_secret|service_role" src` → **소스에 키 하드코딩 없음** ✅
- `git grep "<프로젝트ID>" src` → **URL 하드코딩 없음** (환경변수로만) ✅

> 결론: 저장소가 public이어도 **비밀값은 코드에 없음**. publishable 키가 브라우저에 보이는 건 설계상 정상이며, 실제 보호는 RLS가 담당(3번).

---

## 3. RLS(행 수준 보안) — 라이브 DB 실제 테스트

공개 키(`publishable`)로 직접 접근을 시도해, 허용/차단이 의도대로인지 확인:

| 시도 | 기대 | 실제 결과 |
|------|------|-----------|
| `caregivers` 조회 | 허용 | ✅ 8명 조회됨 |
| `requests` 조회 (연락처 등) | 차단 | ✅ 0건 — **개인정보 노출 안 됨** |
| `caregivers` 삭제 | 차단 | ✅ 8→8명, 삭제 안 됨 |
| `caregivers` 수정 | 차단 | ✅ 수정 반영 안 됨 |

정책 요약 (`supabase/setup.sql`):
- `caregivers`: SELECT·INSERT 허용 / UPDATE·DELETE **정책 없음 = 차단**
- `requests`: INSERT만 허용 / SELECT 없음 = **연락처 공개 조회 차단**

> "공개 키 + RLS 켜짐 = 안전"을 실제로 확인. (RLS 꺼진 채 방치하면 누구나 데이터 조작 가능 — CVE-2025-48757 사례. 본 프로젝트는 해당 없음.)

## 5. 알려진 한계 (정직한 기록)

과제 범위상 허용되나, 실서비스라면 보완할 지점:
1. **등록/요청 INSERT가 공개(anon)** — 공개 폼이라 의도적이지만, 로그인·검증이 없어 스팸 입력이 가능. 실서비스는 인증·검증(예: 자격 서류 확인, reCAPTCHA) 추가 필요.
2. **입력 내용 심층 검증 없음** — 서버 검증은 형식 수준(NOT NULL, 숫자 범위)까지. 값의 진위(실제 자격 보유 여부 등)는 검증하지 않음.
3. **연락처는 저장은 되나 조회는 막힘** — 실제 매칭 운영 시 관리자 전용 조회(인증된 role) 정책이 추가로 필요.
