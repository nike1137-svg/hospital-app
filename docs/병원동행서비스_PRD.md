# 병원동행 매칭 서비스 — PRD (제품 요구사항 정의서)

- **작성일**: 2026-07-16
- **작성자**: 조경호
- **문서 버전**: v0.1 (초안)
- **한 줄 정의**: *병원 방문이 어려운 어르신·보호자가 "우리 동네에서 · 믿을 만한 경력·자격의 · 요양보호사"를 헤매지 않고 찾아 매칭받을 수 있는, 병원동행 요양보호사 매칭 서비스.*

---

## 1. 배경 & 문제 정의

### 1.1 문제
병원 동행이 필요한 상황에서 어르신·보호자는 이런 불편을 겪는다.
- 혼자 병원 가기 어려운 어르신이 **누구에게 동행을 부탁할지** 막막하다.
- 보호자는 직장·거리 때문에 **매번 동행하기 어렵다.**
- 요양보호사를 구하더라도 **경력·자격·지역이 맞는 사람인지** 확인이 어렵다.
- 반대로 요양보호사는 **일감을 어디서 구할지** 마땅한 창구가 없다.

### 1.2 해결 방향
"요양보호사를 나열"만 하는 게시판이 아니라, **지역·경력·자격 조건으로 수요자와 공급자를 이어주는 "매칭" 서비스**로 만든다.

---

## 2. 타깃 사용자

병원동행은 **양면(Two-sided) 플랫폼**이다. 공급자와 수요자를 모두 확보해야 매칭이 성립한다.

| 구분 | 우선순위 | 세그먼트 | 대표 상황 | 핵심 니즈 |
|---|---|---|---|---|
| 수요자 | P0 | 보호자 (자녀·가족) | 부모님 병원 진료·검사 동행 | 믿을 만한 사람, 가까운 지역, 간편한 예약 |
| 수요자 | P0 | 어르신 본인 | 정기 진료, 거동 불편 | 친절함, 소통, 안전 동행 |
| 공급자 | P0 | 요양보호사 | 일감(동행 건) 확보 | 내 지역 일감, 조건에 맞는 매칭, 등록 편의 |
| 수요자 | P1 | 요양기관·병원 | 환자 이송·동행 위탁 | 다건 매칭, 검증된 인력 |

> **MVP 집중**: 수요자 P0(보호자·어르신)와 공급자 P0(요양보호사) 두 축의 **1:1 매칭**. 기관 위탁(P1)은 후속.

---

## 3. 핵심 컨셉 & 차별점

1. **양면 매칭** — 요양보호사 등록(공급)과 사용자 등록(수요)을 조건으로 연결하는 것이 서비스의 본질.
2. **신뢰 정보 우선** — 자격 유무·경력·추가자격증을 목록 전면에 노출해 "믿고 고를 수 있게" 한다.
3. **지역 기반** — 병원동행은 이동이 핵심이므로 "우리 동네" 매칭을 최우선으로 한다.

---

## 4. 주요 기능 요구사항

### 4.1 요양보호사 등록 (공급자)
- 요양보호사가 프로필을 등록한다: **이름, 자격 유무, 경력, 주소지, 연령대, 추가자격증**.
- 프로필 사진 등록(추후 `next/image`로 최적화).
- 자격 미보유(교육 수강 중)는 "예비" 상태로 구분 표시.

### 4.2 사용자 등록 (수요자)
- 보호자·어르신이 동행 요청 정보를 등록한다: 지역, 희망 일시, 진료 목적, 요청 조건(성별·경력 등).

### 4.3 매칭 결과 (최우선 핵심)
- 등록된 수요자 조건에 맞는 **요양보호사 리스트**를 보여준다.
- 정렬·필터: **지역 우선 → 경력 → 자격/추가자격**.
- 카드에 자격 유무·경력·지역·연령대·추가자격을 요약 표시.
- 상세: 프로필 전체 정보 + 동행 요청(연결) 버튼.

### 4.4 관리자 기능 (운영용)
- 요양보호사 자격 검증·승인, 회원 관리, 매칭 이력 확인.

## 5. 기술 스택 (확정 방향)

| 영역 | 선택 | 비고 |
|---|---|---|
| 프론트엔드 / 화면·로직 | **Next.js (React)** | 자료·예제가 압도적으로 많아 막혔을 때 검색 해결 확률이 높음. 매칭 UI를 제약 없이 구현 |
| SEO / 검색 유입 | Next.js SSR·SSG 기본 지원 | **지역·자격 기반 요양보호사 검색 유입**에 유리 |
| 이미지 | next/image 내장 | **요양보호사 프로필 사진** 최적화 |
| 백엔드 (DB·인증·저장) | **Supabase** (후속 단계 도입) | 로그인·프로필/매칭 저장·이미지 저장 |
| 배포 | Vercel | Next.js와 궁합, 무료 배포 가능 |

> **설계 원칙**: 백엔드(로그인·저장·매칭 로직)는 **2단계에서 추가**한다. 1단계는 프론트+매칭 화면 중심으로 만들되, 데이터 구조를 "나중에 백엔드를 꽂을 수 있게" 설계한다. (현재 목업: `src/data/caregivers.json`)

## 7. 데이터 모델 (초안 — Phase 2 대비)

- **Caregiver(요양보호사)**: id, 이름, `자격유무`, 경력(년), 주소지, 연령대, `추가자격증[]`, 프로필사진, 승인상태
- **User(사용자/수요자)**: id, 이름, 연락처, 주소(지역), 관계(본인/보호자)
- **MatchRequest(동행 요청)**: id, user_id, 희망일시, 병원·진료목적, 요청조건, 상태(대기/매칭완료/종료)
- **Match(매칭)**: id, request_id, caregiver_id, 상태(제안/수락/거절/완료)

> 현재 목업 `caregivers.json` 필드: `id, name, isCertified(자격유무), careerYears(경력), address(주소지), ageGroup(연령대), additionalCertifications(추가자격증)`

## 9. 열린 질문 (다음에 정할 것)

1. **자격 정책** — 자격증 보유자만 매칭 노출할지, 교육 수강 중("예비")도 허용할지.
2. **매칭 기준 우선순위** — 지역 최우선인지, 경력·자격 우선인지.
3. **연결 방식** — 앱 내 매칭 수락인지, 전화·문자 연결만 먼저 넣을지.
4. **검증** — 요양보호사 자격 진위 확인을 어떻게 할지(서류 업로드 등).
5. **모바일 우선 여부** — 어르신·보호자 사용성을 고려한 모바일 우선 설계.
